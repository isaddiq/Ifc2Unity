Chapter: IFC Importer and System Architecture (BIMUniXchange)

Overview
The IFC Importer is a two stage conversion pipeline that transforms Building Information Models (IFC) into a Unity scene with geometry, materials, and metadata. The design separates heavyweight IFC parsing and geometry extraction into a Python stage (IfcOpenShell) and Unity side scene construction into a C# stage. This separation minimizes Unity editor dependencies on IFC parsing while retaining a reproducible, file based interchange (OBJ, MTL, CSV). The importer is implemented as an Editor window with a dedicated processor, a data model layer, and reusable loaders for meshes and metadata. The output includes a complete spatial hierarchy and per element metadata that can be queried at runtime.

System Architecture
The architecture is organized as a pipeline with clear module boundaries:
1) Editor UI (IfcImporterWindow.cs): User entry point. Collects file path and options, runs the import, and displays progress and statistics. Also writes a structured log file in Assets/BIMUniXchange/Logs.
2) Import Orchestrator (IfcImportProcessor.cs): Coordinates Python execution, loads exported files, builds the Unity hierarchy, and computes statistics.
3) Data Models (IfcDataModels.cs): Defines IfcElementData, spatial hierarchy nodes, and IfcImportStatistics. These structures are the shared contract across parsing and scene build stages.
4) Metadata Component (IfcMetadata.cs): Serializable MonoBehaviour that stores IFC identity, hierarchy, classification, material info, and full property sets on each GameObject.
5) CSV Parser (CsvMetadataParser.cs): Reads the exporter CSV, creates IfcElementData objects, and builds a storey focused spatial structure for grouping and statistics.
6) OBJ/MTL Loader (ObjMeshLoader.cs): Parses the combined OBJ and MTL, supports multi material submeshes using groups, generates UVs if absent, and builds Unity materials compatible with URP or Standard shaders.
7) Python Exporter (ifc_to_unity_export.py): Uses IfcOpenShell to filter elements, extract geometry and materials, and generate OBJ, MTL, and metadata CSV.

IFC to Unity Workflow
The workflow is intentionally deterministic and file based, so every step can be audited and repeated. The following sequence reflects the actual call chain and data flow.

Step 1. Editor level initialization and validation
- The user opens the Editor window via Window/BIMUniXchange/IFC Importer (IfcOpenShell).
- The UI supports drag and drop of an IFC file and manual browsing of both IFC file path and Python executable path.
- The Test Python action executes the configured interpreter and validates the availability of IfcOpenShell.
- Import options are collected into IfcImportOptions (hierarchy, metadata, materials, colliders, scale, UVs, transparency, shader name). These options are passed to IfcImportProcessor for execution.

Step 2. Python conversion kickoff and monitoring
- IfcImportProcessor.RunPythonConversion resolves absolute paths for both IFC input and exporter script.
- The script path is discovered by checking standard locations under Assets and by searching the AssetDatabase.
- A Python process is spawned with stdout and stderr captured. Lines that contain "Processing:" or "Metadata:" are parsed for progress reporting.
- The conversion phase writes three files next to the IFC: a combined OBJ, a matching MTL, and a CSV metadata file.

Step 3. IFC product filtering in Python
- The exporter loads the IFC file with IfcOpenShell and inspects the schema, project, site, and building metadata.
- Elements are collected using ifc_file.by_type('IfcProduct') and filtered with a two tier logic:
  a) Exclude non physical or helper types (e.g., openings, annotations, grids, ports, feature elements).
  b) Include an allow list of common physical classes (walls, slabs, beams, MEP, furniture, and many others).
  c) If an element is not in the allow list, include it if it has exportable geometry representations (Body, Brep, SurfaceModel, Tessellation, or mapped representations), or if any representation item indicates valid geometric types.
- Decomposed children are explicitly added, ensuring sub elements (e.g., railing parts) are included if they have geometry.

Step 4. Metadata extraction and spatial hierarchy derivation
- For every element, get_element_metadata assembles identity and descriptive fields (GlobalId, Name, IfcClass, Description, ObjectType, Tag).
- Type information is retrieved via ifcopenshell.util.element.get_type, stored as TypeName and TypeId.
- Spatial hierarchy is computed by get_spatial_hierarchy using containment (ContainedInStructure), references (ReferencedInStructures), and decomposition (Decomposes). The algorithm walks upward to resolve Project, Site, Building, Storey, and Space and formats names with explicit class prefixes (e.g., "IfcBuildingStorey: Level 1").
- Property sets and quantity sets are flattened into key value pairs with prefixes such as Pset_<PsetName>_<Property> and Qto_<QtoName>_<Quantity>. These are appended to the metadata row.

Step 5. Material extraction and color normalization
- MaterialManager resolves materials via multiple strategies, prioritizing styled items and element material associations, then falling back to type level materials.
- Colors are read from IfcSurfaceStyleRendering or IfcSurfaceStyleShading when present, and transparency is inverted to alpha.
- Material names are sanitized for OBJ and MTL compatibility, then stored in a material dictionary used later during MTL export.

Step 6. Geometry extraction with multi material support
- For each product, process_geometry_multi_material uses IfcOpenShell geometry settings with world coordinates enabled and no vertex welding.
- Vertices are transformed from IFC coordinates to Unity coordinates with a 180 degree Y rotation: x = -x, y = z, z = -y. Normals are transformed consistently.
- Faces are grouped by material_id to create submeshes. Each submesh stores triangles and a material reference, enabling per face material assignment in Unity.
- If geometry creation fails for an element, the script logs the error but still records metadata to keep the CSV complete.

Step 7. Export of interchange files
- write_combined_obj_multi_material produces a single OBJ that contains all elements. Each element is written as an object (o), and each material region inside the element is written as a group (g) with a usemtl line.
- write_mtl exports all materials with diffuse color, transparency, and shininess.
- write_metadata_csv outputs a CSV with stable priority columns followed by all discovered properties. This CSV is the source of truth for metadata in Unity.
- Spatial hierarchy elements (IfcProject, IfcSite, IfcBuilding, IfcBuildingStorey) are appended as metadata only records so the Unity hierarchy can be created even if the IFC did not include explicit project level geometry.

Step 8. Unity side loading of exported files
- LoadExportedFiles resolves file paths based on the IFC name and reads the CSV into IfcElementData objects. The CSV parser preserves every value, including empty fields, and tracks counts for statistics.
- CsvMetadataParser.BuildSpatialHierarchy builds a site building storey structure by grouping elements by their stored hierarchy strings and assigns unclassified elements into a default site and building.
- ObjMeshLoader.LoadObjFileMultiMaterial parses the combined OBJ, builds a Unity Mesh for each element, uses 32 bit indices to avoid vertex limits, and preserves groups as submeshes. When UVs are missing, planar UVs are generated.
- ObjMeshLoader.LoadMtlFile creates Unity materials, applies colors and transparency, and supports both URP Lit and Standard shaders.

Step 9. Scene assembly and metadata attachment
- BuildSceneHierarchy creates a root GameObject, then constructs Project, Site, Building, and Storey nodes using metadata. Defaults are created if any level is missing.
- Each non spatial element is parented under its storey node (or a default storey). This preserves a building decomposition that mirrors IFC containment.
- CreateElement attaches MeshFilter, MeshRenderer, and optional MeshCollider. For multi material meshes it assigns a materials array by submesh. For single material meshes it uses the MTL material or a color based fallback.
- AssignMetadata attaches an IfcMetadata component with identity, spatial path, classification, material, color, and the full property list from the CSV. This makes the Unity scene semantically searchable.

Data Loss Measurement System
The importer provides a structured measurement of data retention during the pipeline.
- Baseline counts are taken from the CSV metadata file (CsvElementCount). The CSV includes both spatial hierarchy records and all product elements, regardless of whether their geometry was successfully created.
- SuccessfullyImported counts the number of non spatial elements that became GameObjects. Elements without a matching mesh still count as imported but are tracked separately as LogicalNodes when geometry is absent.
- DataLossPercentage is computed as (CsvElementCount - SuccessfullyImported) / CsvElementCount * 100. This is reported in the log as data loss and includes the absolute number of lost elements.
- MeshMatchRate compares ElementsWithGeometry to ObjMeshCount. MeshesNotFound counts elements that had metadata but no mesh in the OBJ.
- Additional diagnostics include elements with color data, per class counts, per storey counts, timing per phase, and memory usage snapshots.

This system is designed for repeatable, file based audits. Because the CSV includes spatial hierarchy elements, the reported data loss can reflect those spatial records rather than actual geometry loss. For example, a model with 1 project, 1 site, 1 building, and 6 storeys will produce 9 metadata records that are not counted as SuccessfullyImported, which appears as a small data loss even when all physical elements are imported.

Limitations and Challenges
1) Option coverage: Several UI options (GroupByStorey, GroupByClass, ScaleFactor, OptimizeMeshes, GenerateUVs) are present in the Editor window and options struct but are not fully applied in the current processor logic. The scene is always grouped by storey, and scale is not applied to geometry.
2) Timing aggregation: TotalTimeSeconds is measured only during scene assembly, not the full pipeline. This can cause total time values in logs to be smaller than mesh load or Python conversion times, which should be interpreted as phase level metrics rather than end to end time.
3) Data loss metric interpretation: The data loss formula mixes spatial metadata entries with physical elements. This can over report loss in models with many hierarchy nodes. A refined metric should exclude spatial elements or report separate counts.
4) Metadata key schema: The exporter writes property keys using underscores (e.g., Pset_WallCommon_IsExternal). The C# metadata component accepts any key, but any downstream tooling that expects dot separated keys may require normalization.
5) Coordinate conversion constraints: The exporter applies a fixed IFC to Unity axis conversion with a 180 degree Y rotation. This is correct for standard IFC coordinates but may not match non standard coordinate systems or authoring tool conventions without additional verification.
6) Large model performance: OBJ and CSV are text heavy and can grow rapidly with large IFC files. Memory usage can spike during mesh loading and CSV parsing due to full file reads. The log example shows multi gigabyte memory changes for large models.
7) Geometry coverage: Elements without representations, excluded types, or failed IfcOpenShell shape generation are kept in the CSV but lack geometry, resulting in logical nodes without meshes. This is expected but should be documented when completeness of geometry is required.
8) Unicode and log readability: The log output contains decorative characters in the Editor window that do not round trip cleanly in all encodings, which can reduce log readability when viewed outside the editor.

Conclusion
The IFC Importer implements a practical and auditable BIM to Unity pipeline using a Python exporter and a Unity side loader. The architecture supports rich metadata, multi material geometry, and a spatial hierarchy aligned with IFC semantics. The current implementation prioritizes robustness and traceability, while exposing clear areas for refinement in timing aggregation, option coverage, and data loss measurement.
